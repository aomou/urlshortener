## Project Overview

A Django-based URL shortener service with OAuth login (Google) and click tracking analytics. This is an MVP project following the "Thin Models, Thin Views, Fat Services" architecture pattern.

**Tech Stack:**
- Django 6.0.1 with Django Templates
- PostgreSQL database
- django-allauth for OAuth integration
- django-user-agents for User-Agent parsing
- Sqids for URL encoding
- uv for Python package management

## Architecture

### Service-Oriented Design

This project follows a strict separation of concerns with business logic in service layers:

**Service Layer (`services.py` in each app):**
- Contains all business logic
- Handles multi-table operations
- Integrates external libraries (Sqids, User-Agent parsing)
- Defines and raises domain-specific exceptions

**View Layer:**
- Handles HTTP request/response
- Performs authentication/authorization checks
- Calls service layer methods
- Catches service exceptions and converts to HTTP responses
- Should NOT contain business logic

**Model Layer:**
- Defines database schema
- Basic field definitions and relationships
- Should NOT contain business process methods

### Key Services

**URLService** (`shortener/services.py`):
- `create_short_url(user, original_url)` - Creates short URL with Sqids encoding
- `get_url_by_code(code)` - Decodes and retrieves URL by short code
- `get_user_urls(user)` - Returns all URLs for a user

**AnalyticsService** (`shortener/services.py`):
- `record_click(url, request)` - Records click with IP anonymization and User-Agent parsing
- `get_url_stats(url_obj)` - Returns aggregated statistics and click logs

### Data Models

**URLModel** (`shortener/models.py`):
- `short_code` - Unique code generated from user_id + url_id using Sqids
- `original_url` - The long URL to redirect to
- `user` - ForeignKey to Django User
- `created_at` - Timestamp

**ClickLog** (`shortener/models.py`):
- `url` - ForeignKey to URLModel
- `clicked_at` - Timestamp
- `ip_address` - Anonymized IP (last octet masked)
- `browser`, `os`, `device_type` - Parsed from User-Agent
- `referer` - HTTP Referer header

### URL Routing Structure

```
/                    - Landing/login page (public)
/accounts/*          - OAuth flow (django-allauth)
/my-urls/            - User's URL list + create form (login required)
/shorten/            - POST endpoint to create short URL (login required)
/stats/<code>/       - Statistics detail page (login required, owner only)
/<code>/             - Short URL redirect (public, 302 redirect)
```

**Important:** The catch-all `/<code>/` route must be defined LAST in URL patterns to avoid intercepting other routes.

## Implementation Requirements

### Sqids URL Encoding

Short codes are generated by encoding `[user_id, url_id]` with Sqids:
- Minimum length: 6 characters
- Reversible: decode short_code to get original IDs
- Ensures different users get different codes for same URL

### OAuth Configuration

django-allauth must be configured with:
- `django.contrib.sites` in INSTALLED_APPS
- `SITE_ID = 1`
- Custom AUTHENTICATION_BACKENDS including allauth
- Social Applications created in Django Admin with Client ID/Secret
- Login redirect: `/my-urls/`
- Logout redirect: `/`

### Click Tracking Flow

When a short URL is accessed:
1. View receives short code from URL
2. Call `URLService.get_url_by_code(code)` to get URL object
3. Call `AnalyticsService.record_click(url, request)` to log the click
4. Return 302 redirect to original URL

**Why 302 (not 301)?** Temporary redirects ensure browsers don't cache, so every click goes through the server for tracking.

### IP Address Handling

`AnalyticsService._get_client_ip` resolves the real client IP with three-tier fallback:
1. `CF-Connecting-IP` ‚Äî set by Cloudflare; cannot be spoofed by upstream clients
2. `X-Forwarded-For` ‚Äî for non-CF reverse-proxy setups (first IP wins)
3. `REMOTE_ADDR` ‚Äî direct connection / local dev

Stored full in DB for audit; anonymized in UI by masking the last octet (e.g., `192.168.1.100` ‚Üí `192.168.1.0`).

## Error Handling

Services raise domain-specific exceptions that Views must catch:

**Common Exceptions:**
- `UrlNotFoundError` - Invalid or non-existent short code ‚Üí 404
- `ValidationError` - Invalid URL format ‚Üí Form error message
- `AccessDeniedError` - User not owner ‚Üí 403

Views should never let service exceptions bubble up; always wrap service calls in try/except.

## Configuration Notes

### Environment Variables

Loaded from `.env` (dev) or `.env.production` (prod). See `.env.example` for the full list.

- `SECRET_KEY` ‚Äî Django secret key
- `DEBUG` ‚Äî `False` in production
- `SITE_DOMAIN` ‚Äî production domain (e.g., `example.com`); appended to `ALLOWED_HOSTS` and `CSRF_TRUSTED_ORIGINS`. Leave blank in local dev.
- `DATABASE_URL` ‚Äî optional; takes precedence over individual `DB_*` vars
- `DB_NAME` / `DB_USER` / `DB_PASSWORD` / `DB_HOST` / `DB_PORT` ‚Äî Postgres connection (host is `db` inside Docker Compose)
- `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` ‚Äî OAuth credentials (passed to allauth via `SOCIALACCOUNT_PROVIDERS`, no Django Admin step needed)

### Database

PostgreSQL in both development and production (`django.db.backends.postgresql`). Configured via `DATABASE_URL` (using `dj_database_url`) with fallback to individual `DB_*` vars.

### Static Files

WhiteNoise handles both pipeline and serving:
- **Build-time**: `CompressedManifestStaticFilesStorage` runs during `collectstatic` (in the Dockerfile) to produce hashed filenames + pre-compressed `.gz`/`.br` files in `/app/staticfiles/` inside the image
- **Runtime**: `WhiteNoiseMiddleware` serves `/static/` directly from the image filesystem, including the pre-compressed variants
- Host nginx does not handle `/static/`; everything proxies to gunicorn and WhiteNoise short-circuits before the view layer

Trade-off: WhiteNoise is ~10x slower than nginx for static serving, but throughput is far above what this app needs. The simplification (no static volume bind-mount, no permissions juggling) is worth it.

### Reverse Proxy

`SECURE_PROXY_SSL_HEADER = ("HTTP_X_FORWARDED_PROTO", "https")` and `USE_X_FORWARDED_HOST = True` are set so Django correctly detects the original `https` scheme behind Nginx + Cloudflare.

### Deployment

VPS-based deployment: Docker Compose runs web + db; **system-level nginx** on the host acts as the public reverse proxy (TLS termination with Cloudflare Origin Cert at Full Strict, real IP from `CF-Connecting-IP`). Web container only listens on `127.0.0.1:8000`. See `doc/deployment.md` for the full step-by-step guide.

## Testing Strategy

Focus tests on:
- Service layer business logic (URL creation, Sqids encoding/decoding, analytics)
- View authorization (owner-only access to stats)
- Edge cases (invalid short codes, malformed URLs)
- IP anonymization correctness

Models are thin, so minimal model testing needed.
